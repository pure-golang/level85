# AGENTS.md

Стратегический контекст для агентов (инварианты). Тактические инструкции вынесены в `.agents/skills/`. Перед работой с конкретной областью — загружай соответствующий скилл.

## Architecture Principles

### Package Structure
- `doc.go` — контракт пакета, **читать первым** при работе в директории

### Testing Layers
Проект выделяет четыре слоя тестирования + smoke, каждый со своим расположением и инструментом:

| Слой | Расположение | Инструмент | Назначение |
|---|---|---|---|
| unit | `*_test.go` рядом с кодом | Go testing | алгоритмы, чистые функции, компоненты с моками |
| integration | `test/integration/` | testify/suite + testcontainers | технический контракт адаптеров/репо |
| bdd | `test/bdd/steps/` + `test/bdd/features/` | godog + testcontainers | бизнес-сценарии, видимые аналитику |
| e2e | `test/e2e/` (`//go:build e2e`) | Go testing | интеграция с реальными staging-сервисами |
| smoke | `test/smoke/` (`//go:build smoke`) | testcontainers/compose | liveness/readiness собранного стека |

- Shared setup (testcontainers bootstrapping) вынесен в экспортируемый пакет `test/support/`, импортируется из integration/bdd/e2e. Подробнее — `x-integration-testing`.
- `.feature` файлы физически живут в `test/bdd/features/`, из `docs/features` положен symlink на реальную директорию для навигации аналитиков. Подробнее — `x-bdd-godog`.

### Import Isolation (internal/ и pkg/)
- **Leaf-пакеты (`repo`, `storage`, `rabbitmq` etc.) НЕ импортируют друг друга**
- Разрешённые импорты для leaf-пакетов: `domain`, `dto`, `config`, собственные под-пакеты
- **Service-пакеты** зависят от leaf-пакетов только через интерфейсы — НЕ конкретные реализации
- **Единственная точка сборки** — `cmd/*/main.go`: только там импортируются все пакеты и собирается граф зависимостей
- Эталон: `../calls/internal/` — ни один leaf не знает о соседях

### Constructor Rules
- **Конструктор возвращает одно значение — никогда не ошибку**: `func New(cfg Config) *Adapter`
- Имя конструктора — `New`, если в пакете один тип; `New<Name>` — если типов несколько: `func NewReader(...)`, `func NewWriter(...)`
- `Config` передаётся по значению (не указатель) — намеренное копирование, чтобы вызывающая сторона не могла мутировать конфиг после создания
- Конструктор только сохраняет конфиг и инициализирует лёгкие поля (без I/O, без соединений)
- Ошибки соединения/инициализации откладываются до `Start()`, `Connect()` etc.
- Если объект реализует `Close()` — `defer` регистрируется сразу после `New`, безусловно. Ошибку `Close` всегда обрабатывать: либо подменять возвращаемый `err` (если он nil), либо логировать

## Global Conventions

- **Не исправять файлы автогенерации go**
- **log messages и error messages: English** 
- **Строчная первая буква в error messages** (исключение: аббревиатуры большими буквами)
- **Заглавная первая буква в log messages и комментариях**
- **Все комментарии и документация: на русском** 
- Doc-комментарии (`//` перед функцией/типом/пакетом) — полное предложение, заканчивается точкой
- Инлайн-комментарии внутри тела функций — короткая фраза, без точки
- Всегда оборачивать ошибки с контекстом: `fmt.Errorf("failed to ...: %w", err)`
- Всегда использовать `errors.As()` для type assertion, никогда прямой каст
- Никогда не игнорировать ошибки (`_ = err` запрещено)
- Все функции и методы должны начинаться с глаголов
- **Import alias для адаптеров: префикс `a` + краткое имя**: `apgx`, `aenv`, `ahttp`, `aqueue`, `armq`, `alogger`

### Тесты должны падать, а не скипаться

- **`t.Skip()` запрещён** — единственное исключение: `testing.Short()` как маркер интеграционного теста
- Если тест падает из-за отсутствия env-переменных — **настрой окружение**, а не пропускай тест
- Никогда не «чинить» падающий тест через `t.Skip` / `t.Skipf` — это маскирует проблему
- `t.Fatal` / `t.Fatalf` — допустимый способ сигнализировать об ошибке setup

### Контракт важнее реализации

Например, если функция возвращает `error` — обрабатывай, даже если текущая реализация всегда возвращает nil. Реализация — деталь, контракт — обещание API.

## Agent Roles

- Роли агентов проекта находятся в `.agents/agents/`
- Если пользователь даёт короткую команду вида `исполни X`, `запусти X`, `сделай X` и `X` похоже на имя роли — сначала ищи подходящий файл роли в `.agents/agents/`
- Короткое имя роли `X` следует резолвить в файл `.agents/agents/X.md`, если такой файл существует
- Если в `.agents/agents/` несколько подходящих файлов и выбор неоднозначен — запроси уточнение
- Файлы ролей в `.agents/agents/` не следует загружать в контекст целиком заранее
- Для discovery и выбора роли сначала читай только frontmatter файла роли и используй его как краткое описание назначения, ограничений, списка skills и ожидаемых инструментов
- Полный текст файла роли следует читать только после того, как агент решил реально исполнять эту роль
- При исполнении роли агент обязан прочитать полный текст файла роли и загрузить все skills, перечисленные в секции `skills:`
- Правило загрузки ролей должно быть таким же ленивым, как и для skills: сначала метаданные для выбора, затем полный текст только для применяемой роли

## Restricted Skills

Эти скиллы **запрещено запускать самостоятельно** — только по явной команде пользователя:

| Скилл | Разрешённые триггеры |
|---|---|
| `x-commit` | «закоммить», «сделай коммит», «/commit» |

Даже если задача завершена и изменения готовы — не коммить без команды.

## Definition of Done

Перед тем как считать задачу выполненной — запусти субагенты последовательно или параллельно (на твоё усмотрение):
- `dod-reviewer` — проверяет код, документацию, наблюдаемость
- `test-reviewer` — проверяет тесты
- `bdd-reviewer` — **только если затронут BDD-слой** (изменения в `test/bdd/**` или `docs/features`). При отсутствии изменений агент сам завершится сообщением «BDD-слой не затронут».
