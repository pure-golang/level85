# AGENTS.md

Стратегический контекст для агентов: только инварианты репозитория, маршрутизация и ownership правил.

Тактические инструкции живут в `.agents/skills/`. Перед работой с конкретной областью загружай соответствующий скилл и не дублируй его содержание в корневых соглашениях.

При создании или редактировании локальных skills загружай `skill-creator` вместе с `x-skill-bundles`: первый задаёт общий процесс конструирования skills, второй — project-specific требования к skill bundles в `level85`.

`.claude/` в этом проекте — Claude-specific символьная ссылка для совместимости, а не отдельное дерево файлов. Для discovery, чтения и редактирования skills используй `.agents/` как основной источник правды, если текущая модель не из семейства Claude.

## Инварианты репозитория

### Структура пакетов

- `doc.go` — контракт пакета; читай его первым при входе в директорию пакета
- leaf-пакеты (`repo`, `storage`, `rabbitmq` и т.п.) не импортируют друг друга
- допустимые импорты leaf-пакета: `domain`, `dto`, `config`, собственные под-пакеты
- service-пакеты зависят от leaf-пакетов только через интерфейсы, а не через конкретные реализации
- единственная точка сборки графа зависимостей — `cmd/*/main.go`

### Конструкторы и lifecycle

- конструктор возвращает одно значение и не делает I/O
- `Config` передаётся по значению
- тяжёлая инициализация и подключения откладываются до `Start()`, `Connect()`, `Run()` и подобных методов
- если объект реализует `Close()`, cleanup регистрируется сразу после `New(...)`, а ошибка закрытия не игнорируется

### Тестовые слои

| Слой | Расположение | Назначение |
|---|---|---|
| unit | `*_test.go` рядом с кодом | алгоритмы, чистые функции, компоненты с моками |
| integration | `test/integration/` | технический контракт адаптеров и репозиториев |
| bdd | `test/bdd/steps/` + `test/bdd/features/` | бизнес-сценарии через godog |
| e2e | `test/e2e/` | интеграция с реальными staging-сервисами |
| smoke | `test/smoke/` | liveness/readiness собранного стека |

- shared test setup для integration/bdd/e2e живёт в `test/support/`
- `.feature` файлы физически живут в `test/bdd/features/`; `docs/features` — symlink для навигации

### Глобальные соглашения

- не исправляй автогенерированные Go-файлы
- log messages и error messages пишутся на английском
- комментарии и документация пишутся на русском
- error messages начинаются со строчной буквы
- log messages и комментарии начинаются с заглавной буквы
- doc-комментарий — полное предложение с точкой
- инлайн-комментарий — короткая фраза без точки
- import alias для адаптеров: префикс `a` + краткое имя

### Терминология проекта

- «частично применяемые интерфейсы» — локальные интерфейсы у потребителя, содержащие только реально используемые методы зависимости; не подменяй этот термин синонимами без явного запроса пользователя

## Владельцы правил

У каждого повторяемого правила должен быть один канонический владелец.

| Тема | Канонический владелец |
|---|---|
| `doc.go`, package contract, doc comments | `x-doc-go` |
| test layers, `t.Parallel()`, `testing.Short()`, AAA, `t.Cleanup` | `x-testing-conventions` |
| BDD layout и нумерация сценариев | `x-bdd-godog` |
| продуктовый путь BDD (`PRD -> .feature`) | `x-bdd-product-workflow` |
| разработческий путь BDD (`red -> green -> blue`) | `x-bdd-dev-workflow` |
| legacy harvest в `.feature` | `x-bdd-knowledge-harvest` |
| env config, `.env`, `env.InitConfig` | `x-env-config` |
| logging policy | `x-log` |
| tracing, metrics, middleware order, monitoring bootstrap | `x-observability` |
| adapter package workflow | `x-new-adapter` |
| PostgreSQL/repo patterns | `x-database-patterns` |
| consumer-side mocks | `x-mockery` |
| test matrix document | `x-test-matrix` |
| Go command scope и Taskfile preference | `x-go-commands` |
| первичная диагностика test/env/service | `x-troubleshooting` |

Если у правила уже есть владелец, в остальных местах оставляй только короткую ссылку, а не пересказ.

## Роли агентов

- роли проекта находятся в `.agents/agents/`
- если пользователь даёт короткую команду вида `исполни X`, `запусти X`, `сделай X` и `X` похоже на имя роли, сначала ищи `.agents/agents/X.md`
- если файлов несколько и выбор неоднозначен, запроси уточнение
- для discovery роли сначала читай только frontmatter и используй его как краткое описание назначения, ограничений, списка skills и ожидаемых инструментов
- полный текст роли читай только после решения реально исполнять эту роль
- при исполнении роли загружай все skills из секции `skills:`

Роль не должна становиться отдельным источником норм. Нормы принадлежат `AGENTS.md` и skills.

## Restricted Skills

Эти skills запрещено запускать самостоятельно и можно вызывать только по явной команде пользователя:

| Скилл | Разрешённые триггеры |
|---|---|
| `x-commit` | «закоммить», «сделай коммит», «/commit» |

Даже если задача завершена и изменения готовы, не коммить без команды.

## Стратегия реализации

Перед началом работы определи тип задачи и выбери стратегию.

### A. TDD-first

Применяй для новой функциональности, bug-fix, расширения API и новых адаптеров.

Минимальный порядок:
1. Определи затрагиваемые пакеты и файлы.
2. Сначала напиши тесты.
3. Для бизнес-задач начни с `.feature` и используй BDD-skills.
4. Зафиксируй red.
5. Доведи реализацию до green.
6. Выполни refactor под зелёными тестами.

### B. Gate

Применяй для рефакторинга, документации, конфигурации, инфраструктуры и style-only изменений.

Минимальный порядок:
1. Внеси изменения.
2. Проверь релевантные существующие тесты и сборку.

## Definition of Done

Перед завершением работы прогоняй reviewer-ролями только те области, которые реально затронуты.

- `dod-reviewer` — код, документация, observability
- `test-reviewer` — тесты и test-landscape
- `bdd-reviewer` — только если затронут BDD-слой

Reviewer-роль проверяет и отчитывается. Она не владеет правилами и не должна дублировать skill bundles.
