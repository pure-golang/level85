# BDD Dev Workflow (RGB)

Скил описывает **разработческую ветку** BDD-first цикла. Вход — готовый `.feature` из `x-bdd-product-workflow`. Выход — зелёные тесты + реализация.

Цикл использует технику **RGB** из TDD (см. `x-testing-conventions`, Шаг 4). В BDD красное/зелёное определяется сценариями, а не юнит-тестами.

## Предусловие

`.feature` файл уже лежит в `test/bdd/features/NN_epic/NN_name.feature`, прошёл чек-лист из `x-bdd-product-workflow`. Если нет — сначала туда.

## Фаза 1: Red (скелет шагов)

**Цель:** все сценарии должны упасть с понятным pending-статусом, чтобы увидеть полный список того, что предстоит реализовать.

### Шаг 1.1: Сгенерировать скелет step-функций

Запусти прогон:
```bash
task bdd:all
```

godog выведет список нераспознанных шагов с готовыми регексами и сигнатурами:

```go
// You can implement step definitions for undefined steps with these snippets:

func userRegistered(email string) error {
	return godog.ErrPending
}

func InitializeScenario(ctx *godog.ScenarioContext) {
	ctx.Step(`^зарегистрирован пользователь "([^"]*)"$`, userRegistered)
}
```

### Шаг 1.2: Перенести скелет в `test/bdd/steps/`

Группируй шаги **по эпикам**, зеркалируя структуру `features/` (`01_auth_steps_test.go`, `02_billing_steps_test.go`). Пока шагов мало, достаточно одного `steps_test.go`.

Преобразуй функции в методы на `scenarioCtx` (см. `x-bdd-godog`).

### Шаг 1.3: Зафиксировать красное

```bash
task bdd:pending
```

Ожидаемый результат — список всех pending-шагов. Запиши его в описание feature-branch как «TODO for this feature».

## Фаза 2: Green (реализация до зелёного)

**Цель:** прогнать каждый сценарий до зелёного, параллельно реализуя код за шагами.

### Стратегия порядка

1. Сначала базовый сценарий `01. ...` — он задаёт основной путь (happy path).
2. Потом альтернативные сценарии (`02`, `03`...) — другие use cases той же функциональности.
3. Corner cases — буквенные варианты базового сценария (`02A`, `02B`...). Буква означает «вариация того же кейса с другим входом», а не отдельный use case. Например: `02. Отказ при неверном пароле`, `02A. Отказ при пустом пароле`, `02B. Отказ при пароле > 128 символов`.

**Правило:** один сценарий до зелёного перед переходом к следующему. Не разбрасывай реализацию.

### Шаг 2.1: Реализовать step-функцию

Пример — перевод pending в реальный код:

```go
// БЫЛО (red)
func (s *scenarioCtx) givenUserRegistered(email string) error {
	return godog.ErrPending
}

// СТАЛО (green)
func (s *scenarioCtx) givenUserRegistered(email string) error {
	if _, err := pg.DB.Exec(`INSERT INTO users (email, created_at) VALUES ($1, NOW())`, email); err != nil {
		return fmt.Errorf("failed to insert user: %w", err)
	}
	return nil
}
```

### Шаг 2.2: Реализовать код, который шаг проверяет

Если `When` шаг вызывает несуществующий endpoint/функцию — напиши её. Этот момент и есть **BDD-first**: реализация рождается из потребности сценария, а не из абстрактной постановки.

### Шаг 2.3: Запустить конкретный сценарий

```bash
go test ./test/bdd/steps/ -godog.name="^01\."
```

Повторять до зелёного.

### Шаг 2.4: Прогнать весь файл и эпик

Перед переходом к следующему сценарию — проверь что не сломал соседние:

```bash
BDD_STORY=01_auth/01_login task bdd:story
BDD_EPIC=01_auth task bdd:epic
```

## Фаза 3: Blue (рефакторинг)

**Цель:** чистый код реализации под сохранением зелёных тестов.

### Что рефакторить

- **Дублирование между шагами** — вынести общие хелперы в отдельный файл `helpers_test.go` в `test/bdd/steps/`.
- **Магические строки** — в константы пакета.
- **Усложнённые setup'ы** — в `test/support/fixtures.go` (если переиспользуемо между integration и bdd).
- **Сам код реализации** — по общим правилам проекта (`x-errors`, `x-log`, `x-observability`).

### Правила рефакторинга

- После каждой правки запускай `task bdd:all` — ни один сценарий не должен стать красным.
- Не меняй `.feature` файлы на этапе blue. Если нашёл проблему в формулировке — это сигнал вернуться в `x-bdd-product-workflow`.
- Не добавляй новые сценарии в blue — это новая работа, открывай новый RGB-цикл.

## Запреты

❌ **Коммит pending-шагов в main** — только в feature-branch. `Strict: true` в `godog.Options` + `bdd-reviewer` ловят.

❌ **Пропускать red фазу** — «я и так знаю что надо реализовать» приводит к неполному покрытию. Скелет с pending показывает все шаги, включая забытые.

❌ **Реализовывать несколько сценариев параллельно** — слишком много незеленого одновременно, сложно понять что сломалось.

❌ **Изменять `.feature` в процессе реализации без продуктового ревью** — ломает разграничение с `x-bdd-product-workflow`.

❌ **Перенумерация сценариев при удалении** — ломает внешние ссылки. Оставь пропуск в нумерации.

❌ **Юнит-тесты вместо BDD** — BDD не заменяет юнит-тесты, они дополняют друг друга. Юнит-тесты проверяют алгоритмы и граничные случаи функций, BDD — интеграционное бизнес-поведение. См. `x-testing-conventions`.

## Интеграция с другими процессами

- **Перед коммитом:** все сценарии зелёные. Команда `x-commit` вызывается вручную пользователем.
- **Перед PR:** запусти `bdd-reviewer` агент — проверит структуру и layout.
- **В CI на PR:** `go test ./test/bdd/steps/...` — полный прогон (если слой не разросся) или селекция по эпикам через `BDD_EPIC`.

## Смежные скиллы

- `x-bdd-godog` — синтаксис и инфраструктура (обязательно)
- `x-bdd-product-workflow` — откуда приходят готовые `.feature`
- `x-integration-testing` — как устроен `test/support`, который шаги используют
- `x-errors`, `x-log`, `x-observability` — правила реализации production кода
- `bdd-reviewer` (агент) — финальная проверка перед PR
