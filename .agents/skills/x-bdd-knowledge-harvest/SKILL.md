# BDD Knowledge Harvest

Скил описывает процесс **reverse-engineering** бизнес-сценариев из существующего legacy-кода и разрозненной документации. Результат — заполненные `test/bdd/features/` для кодовой базы, которая раньше жила без BDD.

## Когда применять

- Репозиторий существует, работает, но `.feature` файлов нет.
- Есть фрагментарные README, комментарии, changelog — но цельного описания поведения нет.
- Нужно понять «что проект делает» на бизнес-уровне перед рефакторингом или передачей команде.

## Не применяй для

- Новых фич — там работает `x-bdd-product-workflow`.
- Документирования технической архитектуры — это `x-doc-go` и ADR.

## Общий принцип

Цель — вытащить **бизнес-наблюдаемое поведение**. Всё что видит только разработчик (SQL-запросы, внутренние API, структуры БД) — это не `.feature`, оставь в `doc.go` или integration-тестах.

Процесс итеративный: один источник → черновик сценариев → следующий источник дополняет → финальный `.feature` проходит чек-лист из `x-bdd-product-workflow` и `x-bdd-godog`.

## Источник 1: Код + `doc.go`

### Шаг 1.1: Прочитать `doc.go` пакета

`doc.go` описывает технический контракт. Из него выводятся **кандидаты на функции**:
- Каждая публичная функция/метод/HTTP-хэндлер, упомянутая в `doc.go`, — потенциальная Feature.
- Описание «Пакет X делает Y» — кандидат на user story.
- Примеры использования в doc-комментариях — готовые Given/When/Then.

### Шаг 1.2: Прочитать публичные API пакета

```bash
grep -n '^func [A-Z]' internal/auth/*.go
grep -n 'func.*http.Handler' internal/api/*.go
```

Каждая экспортированная функция без «очевидно технического» назначения — кандидат. Например:
- `Login(email, password) → token` — функция «Вход пользователя».
- `ResetPassword(email)` — функция «Сброс пароля».
- `encodePasswordHash(pwd)` — **не функция BDD**, это внутренняя деталь, пропустить.

### Шаг 1.3: Прочитать существующие `*_test.go`

Табличные тесты и integration-тесты часто содержат half-written сценарии. Пример:

```go
// existing test
func TestLogin_InvalidPassword(t *testing.T) {
	user := createUser(t, "user@example.com", "correct")
	_, err := svc.Login(ctx, "user@example.com", "wrong")
	require.ErrorIs(t, err, ErrInvalidCredentials)
}
```

→ конвертируется в:

```gherkin
Scenario: 02. Отказ при неверном пароле
  Given зарегистрирован пользователь "user@example.com"
  When пользователь вводит "user@example.com" и неверный пароль
  Then система отвечает ошибкой "неверные учётные данные"
```

**Правило трансляции:**
- `Arrange` → `Given`
- `Act` → `When`
- `Assert` → `Then`
- Имена констант и ошибок → бизнес-язык (`ErrInvalidCredentials` → `"неверные учётные данные"`)
- Базовый тест → `01. ...`; варианты и negative-кейсы → `02`, `02A`, `03`...

## Источник 2: MD-документы в `docs/`

Ищи фрагменты вида:
- «Как работает X» → user story для функции
- «Что делать если Y» → альтернативный сценарий
- «Бизнес-правило: Z» → отдельный сценарий
- FAQ → каждый вопрос-ответ может стать сценарием
- Changelog → исторические решения; «добавили X потому что Y» — указание на бизнес-правило

### Пример

`docs/payments.md`:
> Если пользователь подписан уже месяц, при повторной оплате применяется скидка 10%.

→

```gherkin
Scenario: 03. Скидка лояльности 10% при повторной оплате после месяца подписки
  Given пользователь подписан на сервис более 30 дней
  When пользователь оплачивает следующий период
  Then сумма платежа уменьшена на 10%
```

## Источник 3: Git history

Git log — это кладбище невыписанных сценариев. Два паттерна:

### 3.1: Сценарии из bugfix-коммитов

```bash
git log --grep="fix" --oneline --all
```

Каждый bugfix → сценарий, описывающий регрессию, чтобы баг не вернулся.

Читай не только заголовок коммита, но и body — там контекст «почему сломалось». Это текст будущего сценария.

```bash
git show <sha>
```

### 3.2: Эволюция фичи из feature-коммитов

```bash
git log --follow internal/auth/login.go
```

Последовательность коммитов показывает как менялась логика. Каждое значимое изменение — кандидат на добавочный сценарий.

## Источник 4: Интервью с разработчиком

Когда автоматизированный анализ исчерпан — есть скрытые знания, которые живут только в головах. Протокол:

1. Показать draft `.feature` знатоку области (пользователю).
2. Задать вопросы:
   - «Что здесь не хватает из скрытых правил?»
   - «Какие edge case вы помните, которые не очевидны из кода?»
   - «Какие сценарии ломались в проде?»
   - «Что пришлось объяснять новому разработчику про этот модуль?»
3. Каждый ответ → новый сценарий. Скрытые правила обычно идут как отдельные сценарии с понятным именем, краш-кейсы — как сценарии ошибок.

**Важно:** интервью — единственный источник для действительно скрытых знаний. Не пропускай, если стоимость ошибки высокая.

## Связь с `doc.go`

`x-doc-go` описывает **технический контракт пакета**. `x-bdd-knowledge-harvest` извлекает **бизнес-наблюдаемое поведение**. Они дополняют друг друга, не дублируют.

Явная связь — через соглашение проекта: имя директории эпика соответствует модулю (например, `01_auth` ↔ `internal/auth`). Это навигационная привычка, не формальное правило.

## Порядок извлечения

Рекомендуемая последовательность:

1. **doc.go + публичные API** — быстрый обзор функций.
2. **Существующие тесты** — готовый скелет сценариев, минимум ручной работы.
3. **docs/ MD файлы** — бизнес-правила и edge cases.
4. **Git history** — регрессии и эволюция.
5. **Интервью** — закрытие пробелов.

После каждого источника прогоняй черновик через чек-лист из `x-bdd-product-workflow`.

## Антипаттерны

❌ **Конвертация всех тестов подряд** — юнит-тесты часто проверяют техническую деталь, а не бизнес. Фильтруй: если проверка формулируется на языке бизнеса — в `.feature`; если на языке кода — остаётся как unit/integration.

❌ **Слепое копирование из changelog** — «Added X» без контекста не сценарий. Нужен «как пользователь видит X».

❌ **Пропустить интервью** — большинство ценной инфы живёт не в коде.

❌ **Конвертация при рефакторинге «на ходу»** — делайте harvest **до** рефакторинга, чтобы `.feature` фиксировал текущее поведение как контракт. Тогда рефакторинг = зелёные BDD-тесты.

❌ **Технические детали в шагах** — см. `x-bdd-godog` раздел «Антипаттерны».

## Смежные скиллы

- `x-bdd-godog` — формат и структура (обязательно)
- `x-bdd-product-workflow` — чек-лист готовности `.feature`
- `x-doc-go` — технический контракт пакета
- `x-bdd-dev-workflow` — что делать после того как feature извлечены
