# BDD Product Workflow

Скил описывает **продуктовую ветку** BDD-first разработки — как довести требования до готового `.feature` файла, прежде чем разработчик начнёт писать код. Разработческую ветку (red → green → blue) покрывает `x-bdd-dev-workflow`. Синтаксис и структуру feature-файлов — `x-bdd-godog`.

## Когда применять

- Запуск новой фичи с нуля.
- Переработка существующего функционала, требующая продуктового ревью.
- Формализация требований к фрагментарно задокументированной области.

## Этапы SDLC

### Этап 1: Продуктовая аналитика → `docs/product_analytics/`

**Вход:** сырые данные (метрики, опросы, фидбэк пользователей, конкурентный анализ).  
**Выход:** документ с выводами и формулировкой проблемы. Не влияет напрямую на `.feature`, но задаёт «зачем».

### Этап 2: PRD (бизнес-аналитик) → `docs/prd/`

**Вход:** продуктовая аналитика + постановка от стейкхолдера.  
**Выход:** PRD с разделами:
- Бизнес-цель (измеримая).
- Целевая аудитория и сегменты.
- Пользовательские сценарии верхнего уровня (список функций).
- Успех/метрики.
- Ограничения (регуляторные, бюджетные, временные).

**Важно:** PRD описывает **что** и **зачем**, но не **как**. Технические решения — за рамками.

### Этап 3: Бизнес-интервью → драфт `.feature`

**Вход:** PRD.  
**Выход:** первая версия `test/bdd/features/NN_epic/NN_name.feature` с базовыми сценариями.

**Формат интервью:**
1. Выбрать одну функцию (user story) из PRD.
2. Сформулировать user story блок — «Как... Я хочу... Чтобы...».
3. Спросить: «Что пользователь видит/получает, если всё прошло хорошо?» — это базовый сценарий `01. ...`.
4. Записать в формате Given/When/Then на бизнес-языке (русский текст, английские ключевые слова). Не использовать технические термины (БД, API, endpoint).
5. Повторить для каждой функции.

**Минимальный шаблон драфта:**

```gherkin
Feature: Вход пользователя

  Как зарегистрированный пользователь
  Я хочу войти по email и паролю
  Чтобы получить доступ к своему кабинету

  Scenario: 01. Успешный вход с корректными учётными данными
    Given зарегистрирован пользователь "user@example.com"
    When пользователь вводит корректный логин и пароль
    Then пользователь получает доступ к личному кабинету
```

### Этап 4: Системный анализ → альтернативные сценарии

**Вход:** базовый драфт.  
**Выход:** дополненный `.feature` с сценариями других типов.

Системный аналитик проходит по каждому базовому сценарию и спрашивает:
- Какие поля валидируются? Что происходит при невалидном вводе?
- Какие скрытые бизнес-правила действуют? (например, «вход блокируется после 5 неудачных попыток»)
- Граничные условия (пустое поле, максимальная длина, ноль, отрицательное)
- Отказы внешних систем, таймауты, конфликты

Каждая найденная проверка — новый `Scenario: NN. ...` со следующим номером, либо вариант существующего (`Scenario: 02A. ...`). Нумерация — по порядку добавления, **пропуски разрешены**.

### Этап 5: RFP (Request for Proposal) → сбор вопросов от разработки

**Вход:** расширенный `.feature` с альтернативными сценариями.  
**Выход:** список вопросов от front/back/mobile команд.

Разработка читает `.feature` и задаёт вопросы. Типы:
- **Продуктовые** («а что если пользователь уже залогинен на другом устройстве?»)
- **Технические** («какой формат access token? JWT или opaque?»)

### Этап 6: Разбор вопросов

**Продуктовые ответы** → новые сценарии в `.feature`, либо уточнения существующих.  
**Технические ответы** → архитектурные записи:
- В `doc.go` соответствующего пакета (технический контракт)
- В `docs/adr/` как Architecture Decision Record
- В `.feature` технические детали НЕ попадают.

### Этап 7: Готовность к реализации

Feature готов к передаче в `x-bdd-dev-workflow` когда:
- Файл и директория названы по шаблону (`NN_epic/NN_name.feature`).
- Есть user story блок в `Feature:`.
- Все сценарии пронумерованы, нет дублей.
- Все сценарии прошли ревью аналитика и стейкхолдера.
- Нет открытых продуктовых вопросов.
- Технические вопросы зафиксированы в ADR/doc.go.

## Чек-лист готовности feature к RFP

- [ ] Директория `NN_epic/` создана с осмысленным именем.
- [ ] Файл назван `NN_name.feature` (без дублирования номера эпика).
- [ ] Есть заголовок `Feature:` с кратким именем функциональности.
- [ ] Есть user story блок (3 строки `Как... Я хочу... Чтобы...`) после заголовка.
- [ ] Есть минимум один базовый сценарий `01. ...`.
- [ ] Каждый сценарий пронумерован, номера уникальны в рамках файла.
- [ ] Шаги сформулированы на бизнес-языке (без SQL, HTTP, JSON).
- [ ] Формулировки не содержат технической реализации (название endpoint, имена таблиц, коды ошибок HTTP).

## Антипаттерны

❌ **Писать `.feature` до PRD** — получите противоречивые сценарии без привязки к целям.

❌ **Технические детали в шагах на этапе интервью**:
```gherkin
# ПЛОХО
When POST /api/v1/login возвращает 200
# ХОРОШО
When пользователь получает доступ к личному кабинету
```

❌ **Один сценарий покрывает несколько проверок** — разнесите, каждый сценарий = одна бизнес-проверка.

❌ **Пропустить альтернативные сценарии** — только базовый путь означает что реализация не будет обрабатывать ошибки.

❌ **Закрывать технические вопросы прямо в `.feature`** — ломает разграничение слоёв, усложняет ревью аналитику.

❌ **User story блок отсутствует** — без «Как... Я хочу... Чтобы...» теряется контекст функциональности.

## Смежные скиллы

- `x-bdd-godog` — синтаксис и структура feature-файлов, обязательно к прочтению
- `x-bdd-dev-workflow` — что происходит после готового `.feature`
- `x-bdd-knowledge-harvest` — как применять скил для legacy-функционала
- `x-doc-go` — технический контракт пакета (куда уходят ответы на технические вопросы)
