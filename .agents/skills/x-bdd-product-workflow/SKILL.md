# BDD Product Workflow

Этот skill — **канонический владелец** продуктового пути от требований к готовому `.feature`.

Он не владеет layout, naming и numbering BDD-файлов. За это отвечает `x-bdd-godog`.

## Когда применять

- Планируешь новую фичу до начала кодинга.
- Пересобираешь требования перед крупной переработкой поведения.
- Формализуешь бизнес-правила, которые должны попасть в `.feature` после интервью со стейкхолдерами.

Не применяй для:
- синтаксиса и физической структуры `.feature` → `x-bdd-godog`
- red/green/blue реализации → `x-bdd-dev-workflow`
- reverse-engineering legacy-поведения → `x-bdd-knowledge-harvest`

## Core workflow

### 1. Собери продуктовый вход

Вход обычно состоит из PRD, аналитики, stakeholder prompt или уже существующего списка user stories. Технические детали на этом этапе не фиксируй в `.feature`.

### 2. Проведи бизнес-интервью по одной функции за раз

Для каждой user story:
- сформулируй `Как / Я хочу / Чтобы`
- зафиксируй happy path как первый сценарий
- уточни альтернативы, ограничения и отказы как отдельные сценарии

По каждой функции спроси:
- Что пользователь видит, если всё прошло хорошо?
- Что считается ошибкой ввода?
- Какие скрытые бизнес-правила влияют на исход?
- Какие пограничные случаи важны для бизнеса?
- Что должно произойти при отказе внешней системы?

### 3. Собери черновик `.feature`

- структура файла, naming и numbering берутся из `x-bdd-godog`
- шаги пишутся на бизнес-языке
- технические решения не подмешиваются в сценарии

Минимальный шаблон драфта:

```gherkin
Feature: Вход пользователя

  Как зарегистрированный пользователь
  Я хочу войти по email и паролю
  Чтобы получить доступ к своему кабинету

  Scenario: 01. Успешный вход с корректными учётными данными
    Given зарегистрирован пользователь "user@example.com"
    When пользователь вводит корректный логин и пароль
    Then пользователь получает доступ к личному кабинету
```

Структуру имени файла и сценариев проверяй по `x-bdd-godog`.

### 4. Проведи RFP с разработкой

Разработчики читают `.feature` и возвращают вопросы двух типов:
- продуктовые вопросы возвращаются в `.feature`
- технические вопросы уходят в ADR, `doc.go` или другой инженерный контракт

На RFP:
- собрать вопросы front/back/mobile по готовому `.feature`
- продуктовые ответы внести в сценарии
- технические ответы увести в ADR, `doc.go` или другой инженерный документ

### 5. Передай feature в разработческий цикл

Файл готов для `x-bdd-dev-workflow`, когда:
- в нём закрыты продуктовые вопросы
- согласованы happy path и ключевые альтернативы
- naming и numbering проверены по `x-bdd-godog`

## Короткий чек-лист

- есть ясный user story блок
- happy path выделен первым базовым сценарием
- альтернативы и edge cases вынесены в отдельные сценарии
- в шагах нет HTTP, SQL, JSON и других реализационных деталей
- открытые технические вопросы вынесены из `.feature`

## Антипаттерны

- писать `.feature` раньше, чем понятна бизнес-цель
- смешивать несколько проверок в одном сценарии
- оставлять технические детали в шагах
- закрывать технические вопросы прямо в `.feature`
