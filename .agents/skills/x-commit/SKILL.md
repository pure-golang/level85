# Локальный коммит

> **Этот скилл запускается ТОЛЬКО по явной команде пользователя.**
> Никогда не вызывай его самостоятельно в ходе реализации задачи.
>
> **Формат строго:** `type: subject` — никаких scope в скобках, даже если предыдущие коммиты их содержат.

## Процесс

### 1. Собери информацию об изменениях

```bash
# Статус рабочего дерева
git status

# Проверь, есть ли staged-изменения (index)
git diff --cached --stat
```

**Режим работы:**
- **Есть staged-изменения** → коммитить только их. Diff для анализа: `git diff --cached`. Команду `git add` **не выполнять** — пользователь уже выбрал, что коммитить.
- **Нет staged-изменений** → коммитить все unstaged-изменения. Diff для анализа: `git diff`. Перед коммитом добавить файлы через `git add <конкретные файлы>`.

```bash
# Diff для анализа (выбери нужный по режиму)
git diff --cached   # если есть staged
git diff            # если staged пуст
```

### 2. Составь commit message

Анализируй diff и формируй сообщение на английском по правилам:

**Формат:**
```
<type>: <subject>

<body>
```

**Типы:**
- `feat` — новая функциональность
- `fix` — исправление ошибки
- `hotfix` — срочное исправление в продакшене
- `docs` — документация
- `refactor` — переписывание кода без изменений логики
- `test` — тесты
- `style` — форматирование, lint
- `perf` — оптимизация производительности
- `chore` — всё остальное (конфиги, сборка, рутина)

**Правила subject:**
- На английском, нижний регистр, без точки в конце
- Максимально лаконично: 3–5 слов

**Правила body:**
- Обязателен — всегда присутствует
- На английском, свободный текст
- Объясняет *что* и *почему*, а не *как*
- Одно или несколько предложений

**Пример:**
```
feat: add dead-letter queue

Move failed messages to DLQ after max retries so they can be
inspected and replayed without blocking the main queue.
```

### 3. Добавь файлы (если нужно) и создай коммит

```bash
# Только если staged пуст — добавь конкретные файлы (не git add -A)
git add <конкретные файлы>

# Если staged уже есть — сразу коммить
git commit -m "<type>: <subject>" -m "<body>"
```

### 4. Подтверди результат

```bash
git log --oneline -3
git status
```

Выведи итоговый hash коммита и его сообщение.

## Ограничения

- **Не пушить** на origin — только локальный коммит
- **Не использовать** `git add -A` или `git add .` — только конкретные файлы
- **Не пропускать** хуки (`--no-verify` запрещён)
- Если pre-commit хук упал — диагностируй причину, исправь, создай **новый** коммит (не `--amend`)
- Не коммитить файлы с секретами (`.env`, credentials)
